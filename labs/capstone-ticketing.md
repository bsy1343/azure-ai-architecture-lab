## 시나리오 — "오픈 10분에 모든 것이 걸린 티켓 예매"

**배경**: ㈜파도기획은 인기 아티스트 콘서트 티켓을 온라인 판매하려 한다. 지난 공연 때 티켓 오픈 직후 트래픽 폭주로 사이트가 다운돼 큰 손실을 입었다. 이번에는 AI를 적극 활용해 설계 → 검증 → 운영 전 과정을 견고하게 준비한다. 여러분은 이 회사의 클라우드 아키텍트다.

**비즈니스 요구사항**

| 항목 | 내용 |
| --- | --- |
| 평상시 트래픽 | 일 방문자 5,000명 수준 |
| 티켓 오픈 순간 트래픽 | 평상시 대비 약 200배 폭증, 순간 초당 3,000건 요청 예상 |
| 매진 예상 시간 | 오픈 후 10분 이내 |
| 데이터 민감도 | 결제 정보 처리 (카드 정보 직접 저장 없음, PG사 연동 가정) |
| 월 예산 | 평상시 1,000만원 이내, 오픈 당일은 일시 초과 허용 |
| 가용성 목표 | 오픈 당일 99.95% 이상 |
| 운영 패턴 | 티켓 오픈일 외에는 인프라 대부분 유휴 (평균 2주 간격 오픈) |
| 최종 산출물 | 경영진 보고용 HLD 문서 + 오픈 당일 운영 결과 보고서(포스트모템 포함) |

이 시나리오는 "잠깐 반짝 터지는 트래픽에 어떻게 대비할 것인가"라는, 실무에서 매우 흔한 문제(수강신청, 선착순 이벤트, 명절 기차표 예매 등)를 대표합니다.

## Task 1. 기본 아키텍처 설계
> 연계 이론: [Section 1](../ch1-foundation/01-basics-and-ai-persona.md)

AI에게 아래와 같이 요청해 초안을 도출하세요.
```
너는 Azure 클라우드 아키텍트다. 평상시 일 방문자 5,000명이지만, 티켓 오픈
순간 초당 3,000건까지 트래픽이 200배 폭증했다가 10분 내 매진되는 티켓
예매 서비스의 기본 아키텍처를 제안해줘. VNet/Subnet 구조, 컴퓨팅 방식
(VM/VMSS/Container/Serverless 중 선택과 근거), 캐싱 전략을 포함해줘.
평상시 비용은 최소화하고 오픈 순간에만 폭발적으로 확장할 수 있어야 해.
```
> ✅ **체크포인트**: AI가 "평소엔 축소, 오픈 때만 확장"이라는 비대칭 요구를 반영했는가? 단순히 "VMSS 쓰세요"에서 끝나지 않고 캐싱 계층(Azure Cache for Redis 등)을 제안하는지 확인. 잔여 좌석 조회처럼 반복 읽히는 데이터는 캐시 없이는 DB가 먼저 죽습니다.

## Task 2. 네트워크·보안 설계
> 연계 이론: [Section 2](../ch1-foundation/02-network-security.md) · [Day1 Lab2](./day1-labs.md#lab-2--네트워크보안-설계)

결제 정보를 다루므로 3-Tier 구조 + 최소권한 NSG 정책이 필수입니다.

- PG사(결제대행사) 연동 구간은 **아웃바운드 트래픽만** 필요 — 이를 NSG 규칙에 어떻게 반영할지 고민

> ✅ **체크포인트**: "Data Subnet에 인바운드 접근 가능한 곳이 App Subnet뿐인가?", "PG사 연동을 이유로 불필요하게 넓은 아웃바운드 허용 규칙을 만들지 않았는가?"를 스스로 점검.

## Task 3. 컴퓨팅 스펙 산정
> 연계 이론: [Section 3](../ch1-foundation/03-compute-storage.md)

이번 시나리오의 핵심 난이도. VM 1대가 초당 50건을 처리한다고 가정할 때:
1. 오픈 순간 초당 3,000건 처리에 몇 대가 필요한지 직접 계산 (여유율 1.3 적용)
2. 평상시 트래픽(일 5,000명 → 대략 초당 1건 미만) 처리엔 몇 대면 충분한지도 계산
3. 두 숫자 차이가 매우 클 텐데, VMSS 오토스케일만으로 감당 가능할지, 아니면 "오픈 5분 전 미리 스케일업" 같은 사전 조치(Scheduled Scaling)가 필요할지 AI에게 물어 근거 정리

```
Azure VMSS로 평상시 초당 1건, 티켓 오픈 순간 초당 3,000건까지 급증하는
트래픽을 처리해야 한다. CPU 사용률 기반 오토스케일만으로 이 정도의
순간 폭증을 감당할 수 있을지 평가하고, 필요하다면 시간 기반 예약
스케일링(Scheduled Autoscale)을 함께 사용하는 방안을 제안해줘.
```
> ✅ **체크포인트**: CPU 기반 오토스케일은 "이미 부하가 걸린 뒤에" 반응하는 반응형(Reactive) 방식이라는 한계를 AI가 짚는지 확인. 오픈 시각을 미리 아는 이벤트성 트래픽엔 예약 스케일링을 함께 쓰는 것이 정석.

> 📐 **계산 참고 (검산)**:
> - 오픈 순간: `3,000 ÷ 50 × 1.3 = 78대`
> - 평상시: 일 5,000명 ≈ 초당 0.06건 → **1~2대로 충분**
> - 약 **40~78배** 차이. CPU 오토스케일은 부하 발생 후 5~10분 뒤에야 인스턴스가 추가되는데, 이 시나리오는 그 사이 10분 매진이 끝납니다. → **예약 스케일링(오픈 전 미리 확장)이 필수**라는 결론이 수치로 뒷받침됩니다.

## Task 4. 고가용성·비용 최적화
> 연계 이론: [Section 4](../ch1-foundation/04-cost-ha-dr.md)

- 오픈 당일 99.95% 가용성을 만족하려면 Availability Zone 구성이 필요한지 판단
- 평상시(2주 중 대부분) 인프라를 어떻게 축소해 비용 절감할지 AI로 절감안 도출

```
티켓 오픈일에만 높은 트래픽이 발생하고 나머지 2주는 거의 트래픽이 없는
서비스가 있다. 평상시 비용을 최소화하면서도 오픈일에는 즉시 고성능으로
전환할 수 있는 방안을, Reserved Instance/Spot VM/Consumption 서버리스
관점에서 각각 비교해서 제안해줘.
```
> ✅ **체크포인트**: 예약 인스턴스(Reserved Instance)는 "상시 가동"을 전제로 할인하는 상품인데, 이 서비스는 정반대(간헐적 가동)입니다. AI가 이 모순을 짚고 예약 인스턴스 대신 다른 대안(서버리스, 온디맨드+예약스케일링)을 권장하는지 확인. **AI 산출물을 비판적으로 검증해야 하는 대표 사례**.

> 📌 **보강 사실 (검증)**: 예약 인스턴스는 단일 리전에서 24/7 상시 가동되는 안정적 워크로드에 최대 72% 할인을 주는 상품이라, 2주에 한 번 잠깐 쓰는 이 서비스엔 부적합합니다. 대안으로 **Savings Plan**도 있으나 이는 컴퓨팅(VM/App Service/Functions 등)에만 적용되고 **Azure SQL Database·Cosmos DB·Azure Cache for Redis 같은 비컴퓨팅 서비스에는 적용되지 않습니다** — 이런 서비스는 별도 Reserved Capacity로만 약정 할인이 됩니다. 이 서비스처럼 간헐적·이벤트성 워크로드에는 결국 **Consumption 서버리스 + 온디맨드 + 예약 스케일링** 조합이 가장 합리적입니다.

## Task 7. 모니터링 설계

오픈 당일 딱 10분이 서비스의 전부이므로, 사후 로그 분석보다 **실시간 대응**이 훨씬 중요합니다.
```
티켓 오픈 후 10분 동안이 서비스 성패를 좌우하는 이벤트성 서비스다.
```

## Task 8. 장애 시뮬레이션 및 포스트모템
> 연계 이론: [Section 8](../ch2-verification-ops/08-ops-optimization.md)

아래 가상 장애 시나리오로 원인 분석·포스트모템 작성을 연습하세요.

> **가상 장애 시나리오**: 티켓 오픈 2분 후, VMSS가 예약 스케일링으로 미리 확장은 되어 있었으나 Azure Cache for Redis의 연결 수 제한(Max Connections)에 도달하면서 캐시 조회가 실패하기 시작했고, 모든 요청이 그대로 SQL Database로 몰리면서 DTU/vCore 사용률이 100%에 도달, 결제 단계에서 타임아웃이 대량 발생했다.

```
[위 장애 시나리오를 붙여넣고] 이 장애의 근본 원인과, 재발 방지를 위한
단기·장기 개선안을 제안해줘. 특히 다음 오픈 때는 동일한 문제가 재발하지
않으려면 사전에 무엇을 점검해야 하는지도 정리해줘.
```

> 📌 **참고 (실무 사실)**: 이 장애는 실제로 흔한 캐스케이드 장애 패턴입니다. Azure Cache for Redis는 **티어별로 클라이언트 연결 수 상한**이 있고(예: Standard C1은 최대 1,000 연결), 상한 도달 시 새 연결이 타임아웃으로 실패 → 캐시 미스 → 부하가 DB로 전가됩니다. 근본 원인으로는 (1) 클라이언트가 연결을 재사용하지 않고 매 요청마다 새로 여는 문제(싱글턴 ConnectionMultiplexer 미사용), (2) 티어 용량 부족을 짚을 수 있습니다.
>
> ⚠️ **최신 변경**: Microsoft는 **Azure Cache for Redis 전 SKU의 은퇴(retirement) 일정**을 발표했고, 신규 구성은 **Azure Managed Redis** 사용을 권장합니다. 이 실습의 개념(연결 수 제한·캐스케이드 장애)은 동일하게 적용되지만, 실제 신규 구축 시엔 Managed Redis 여부를 확인하세요. (교재 작성 이후의 변경일 수 있음)
